# 公司文档评论 API 接入范围（单向导入）

**决策时间**：#R7
**状态**：✅ 已确认
**关联大纲**：[返回大纲](../outline.md)

---

## 📋 背景

### 问题/需求
用户希望在将公司 docs 文档拉取到本地并转换为 Markdown 时，能够保留公司系统中的评论内容，用于本地 AI/人协作修改。

### 约束条件
- 优先保证本地协作闭环可用（评论定位、thread、关闭/归档）
- 与公司系统对接应尽量降低复杂度，避免引入双向同步与冲突处理

---

## 🎯 目标

- 支持从公司评论 API 拉取评论并注入到本地 Markdown 协议表示中
- 本地协议不需要完整覆盖公司系统的所有能力（例如 emoji reaction）

---

## 📊 方案对比

| 方案 | 描述 | 优势 | 劣势 | 决策 |
|------|------|------|------|------|
| A | 单向导入（公司 → 本地） | 实现简单；无冲突；满足“拉取并保留评论” | 本地评论无法回写公司系统 | ✅ |
| B | 双向同步 | 可回写；协作闭环更完整 | 冲突处理复杂；权限/鉴权/审计成本高 | ❌ |
| C | 公司 API 作为后端（本地为唯一入口） | 体验一致；可扩展 | 相当于重做客户端；工程量巨大 | ❌ |

---

## ✅ 最终决策

### 选定方案
采用 **单向导入（公司 → 本地）**：
- 通过公司 API 拉取文档评论列表
- 转换为本地 Markdown 评论协议（thread）表示
- 后续本地修改与评论处理（resolve/归档/删除）不回写公司系统

### 同步字段取舍
- **不支持 emoji reaction**（导入时可忽略）
- 作者仅保留 **username**（不强依赖头像、组织信息等）

### 预期效果
- 公司文档可“带评论落地到本地”用于 AI/人协作
- 降低接入复杂度与合规风险
