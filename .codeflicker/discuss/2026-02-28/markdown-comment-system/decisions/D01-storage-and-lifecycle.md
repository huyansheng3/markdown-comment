# 评论存储与生命周期策略（A+C 混合 + thread + 归档）

**决策时间**：#R4
**状态**：✅ 已确认
**关联大纲**：[返回大纲](../outline.md)

---

## 📋 背景

### 问题/需求
需要在 Markdown 文档中实现人类与 AI 的双向评论协作，使 AI 能基于评论修改文档；同时支持评论处理完成后的收敛（关闭/清理），避免对 AI 造成上下文干扰。

### 约束条件
- 尽量基于 Markdown 生态扩展，便于与渲染组件（如 react-markdown）集成
- 评论应可定位到文档具体片段（anchor quote）
- 评论有 thread（多轮回复）
- 评论处理完成后，不应继续干扰 AI（最好从输入侧不可见）

---

## 🎯 目标

- 文档协作过程可记录、可回溯
- 评论在“处理完成”后可收敛，文档尽量回归干净
- 兼顾：解析实现复杂度、Markdown 可读性、与前端渲染集成可行性

---

## 📊 方案对比

| 方案 | 描述 | 优势 | 劣势 | 决策 |
|------|------|------|------|------|
| A | 独立 comment.md / sidecar 文件 | 原文档纯净；评论可独立管理 | 文件关联与分发复杂；编辑体验割裂 | ❌ |
| B | 完全内联：仅用 HTML 注释存所有信息 | 就地可读；实现直观 | 结构化差；解析与演进困难；与渲染耦合 | ❌ |
| C | A+C 混合：frontmatter 存元数据 + 正文内联 block 表达位置/线程 | 结构化与可读性折中；利于渲染与工具链 | 需要处理一致性（孤儿元数据/孤儿 block） | ✅ |

---

## ✅ 最终决策

### 选定方案
采用 **A+C 混合方案**：
- **frontmatter**：存 thread 的结构化元数据（id、状态、作者、时间、anchor 信息等）
- **正文**：使用约定的 comment block（自定义 Markdown 语法或约定的 blockquote 形式）表达评论位置与 thread 展示

thread 规则：
- **一个 thread 绑定一个 anchor quote**（首条评论引用的文本片段）
- thread 内允许人类与 AI 多轮回复

评论生命周期：
- open → resolved
- 支持用户手动关闭
- 当被评论文本发生改动时自动关闭（判定方式允许对空格/换行做规范化后再 hash）

已关闭评论处理：
- **物理删除/归档**（而非仅在渲染层隐藏）
- 允许导出 AI-view（已关闭内容不出现）

### 决策理由
- A+C 在“可读性/结构化/生态兼容/演进性”之间更均衡
- thread 绑定单 anchor quote 显著降低锚定与一致性复杂度
- resolved 后物理删除/归档，满足“对 AI 完全不干扰”的核心诉求

### 预期效果
- 文档协作时评论可就地阅读与渲染
- 评论处理完成后可收敛，文档恢复干净
- AI 输入侧天然减少噪音

---

## ❌ 被否决的方案

### 独立 sidecar 文件
- **否决原因**：编辑体验割裂、分发/引用复杂，且与“就地评论”直觉不一致
- **重新考虑条件**：需要强审计/强合规、或评论量极大导致内联不可维护时
